Lecture Notes

I have no doubt that if you can make it through this lesson you can become a successful programmer using Visual C#. I cannot overstate the following: The most important theories you will ever learn in programming take place in this lesson. At this point in your program, we have to marry the visual aspects of programming with the C# language. This isn't by accident: these are two very different concepts, and understanding their differences will help you choose your career path upon successfully completing your course. Object-oriented programming is arguably the most powerful programming methodology in our modern era. Its flexibility in various industries and programming practices alike make the concept universally accepted on programming teams. However, OOP is high-level programming, and in some cases difficult to grasp.

Computer programming is a different way of thinking; programmers must think about details and work through logic that most people don't have to concern themselves with. When you get frustrated learning this material (and you will—no programmer in the world understood OOP on their first introduction to it), remember what I told you in the beginning of this course: practice makes perfect. Read this chapter, follow the examples, and learn the terminology. Know what a class is, what a property is, what a method is, and how to create these things. Don't trivialize terminology at this level, because learning these concepts now will only allow other concepts to "click" later, and trust me, they will!

In the English language, the term object is defined as "Something perceptible by the senses, especially by vision or touch." I want you to stop reading and over the next 10 seconds or so look around the room and count the number of "objects" in the room that you are in. How many did you get? I'm sure at least 10, potentially 100, but if you didn't lift your head too much and used your book, you easily could have gotten to several thousand or so.

The term object is a very generic term. An object can be a book, pages in a book, a page, words on that page, a single word on the page, or letters in a word. In a room, an object can be a lamp, or the light bulb within a lamp, or the filament within the light bulb within a lamp. If you were asked to "return the object to the store," there are countless items that could be misconstrued for what the correct object is that you are to return! As you can see, an object can be something very general (a book) or something very specific (page 135, paragraph number 2, word number 3, character 2). Conceptually understanding what an object is is very important to understanding OOP, which we will study in great detail in this chapter.

Much like the word object is generic in our everyday usage of it, the object in the term object-oriented programming is also somewhat generic. This is because in OOP the programmer can decide what the object is. The programmer defines the object, how it works, what it does, and what it looks like. The programmer then tells the computer this information, and those directives are then carried out within a program.

Every object in our reality has subtleties and features; similarly, objects in OOP also have features. A lamp has attributes such as width and height, has actions such as turn on and off, and has a general routine purpose, such as to illuminate. These same things can be written for an object in OOP. In OOP, a lamp's attributes are called properties. Much like the properties of the form object, or the Label control (which is also an object) we discussed previously, properties allow the programmer to assign information to an object, such as width and height, along with any information that may be deemed necessary by the programmer. Perhaps the programmer wants to also know the lamp’s manufacturer, model type, serial number, date of purchase, the room the lamp resides in, and whether it is currently on or plugged in; these can all be properties of a lamp object defined by a programmer.

As discussed earlier, a lamp also has actions. In OOP, actions are referred to as methods. If an object is considered a “noun” (a person, place, or thing), then a method is the “verb” in the OOP world, because methods do things. The actions mentioned earlier were a simple example, but are actions nonetheless—turn off and turn on. Using OOP logic, the programmer can create a routine once and reuse it countless times. In the logic, the programmer can also determine the current state of the lamp object by inspecting its properties to determine if it is currently plugged in and currently off or on. Once this is determined, the appropriate logic within the method can be written.

Many details and features can go into something as simple as a lamp. In the OOP world, these features are all defined as a class. A class is the definition of the object that is being programmed. The class contains all the actions and properties and pre-programmed logic you want to create with your object, so you write it once and reuse it countless times. The general purpose of creating classes in OOP is so a programmer can reuse the logic, so other programs can use the classes, and so that the hairy details of how something works are hidden, or encapsulated. To a programmer, simply running a method called "turnLampOn" turns the lamp on. The programmer may have little to no knowledge of the tens, hundreds, or thousands of lines of code required to make this function actually work, it just works. It's simplicity in design.

This lesson goes into great detail on classes, objects, and methods and on how to design a class. It also provides several examples on how to visually document class design using the Unified Modeling Language (UML), which is incredibly handy when designing OOP programs with a team of programmers. And, of course, your assigned reading is also packed full of examples that you can (and should) walk through with the textbook to fully grasp the nuances of class design in OOP. With that being said, let's dive in!
